# DR4.T1 — 헌법 사전 가드 (welding-architect)

**판정**: **NEED_USER_DECISION** (스트리밍 방식 1건은 사용자 결정 필수, 나머지 회색영역은 architect 권고 동반)

## 후보별 헌법 검증

### 후보 1 — 응답 스트리밍 ⚠️ D-001/D-004 충돌
- 현 구조: LLM이 `{reply, new_state, options}`를 **단일 JSON**으로 반환 → `EngineResponseSchema` 검증(D-004 1회 재시도) → 클라가 `splitDiff(prev, new_state)` 계산(D-001).
- 충돌 핵심: **반쪽짜리 스트리밍 JSON은 Zod 검증이 불가**하고, `new_state`가 reply와 같은 객체에 박혀 있어 "검증 후 diff" 원칙(D-001)을 끊을 수 있다. → 스트리밍은 *어떻게 자르느냐*가 곧 헌법 문제. **사용자 결정 필요 (GDR4-1)**.

### 후보 2 — 셰프 제안/되묻기 ✅ 조건부 PASS
- D-003(한 턴 한 단계): 셰프가 모호 입력에 **되묻기**(질문만 하고 new_state 없음)는 "한 턴에 0단계 전진"으로 해석 가능 — 단계를 *건너뛰지 않으므로* D-003 정신 부합. 단 "턴마다 반드시 상태가 바뀐다"는 암묵 가정은 깨짐 → 응답 모드 분기 필요 (GDR4-2).
- D-002(생성=카드, 수정=diff): 제안은 *아직 확정 아님*이므로 diff/카드로 박으면 안 됨 — 제안은 별도 표현(잠정 칩) (GDR4-3).

### 후보 3 — 근거 투명성 강화 ✅ PASS
- D-024 Plan / D-025 Context 메타의 *확장*. 기존 `context_used` 서버 wrapper + StagePlanCard 재사용. schema/엔진 contract 무변 가능. 충돌 없음.

## 회색영역 (사용자 결정 / architect 권고)

### GDR4-1 — 스트리밍 방식 [사용자 결정 필수]
- **A. 클라 의사-타이핑**: 전송은 지금처럼 원자적 POST(검증·diff 그대로). 받은 `reply`를 클라에서 글자 단위로 흘려보임. → D-001/D-004 무손상, 위험 0, 엔진 변경 0. 단 *진짜* 스트리밍 아님(우체통은 이미 다 와있음).
- **B. 2단계 SSE** (권고): `reply`(대화 텍스트)는 토큰 스트리밍, `new_state`(구조)는 **끝에 원자적 1건**으로 검증·전송. 진짜 페어링 감각 + D-001/D-004 보존. 단 /api/recipe 전송 방식(ReadableStream/SSE) 신설 + LLM이 reply→JSON 순서로 내도록 프롬프트 규율 + 재시도는 구조 파트에만 적용 — 엔진 변경 가장 큼.
- **C. 완전 구조 스트리밍**: 부분 JSON/tool-use 스트림. D-001 "검증 후 diff"와 최대 충돌 → architect 비권고.

### GDR4-2 — 되묻기(상태 없는 턴) 허용 — 권고 채택
- EngineResponse에 응답 모드 추가(예: `act` = 상태 전진 / `ask` = 되묻기). `ask`면 new_state 생략 + 질문 + 선택지. D-003 정합(단계 건너뛰지 않음).

### GDR4-3 — 제안 표현 — 권고 채택
- 셰프의 능동 제안은 **잠정 칩**(클릭 시 submit)으로. 확정 전이므로 diff/카드 금지(D-002). 기존 옵션 칩 인프라 재사용.

## 신규 ADR 후보
- **D-029** 스트리밍 정책 (GDR4-1 결정에 종속)
- **D-030** 응답 모드 act/ask — 되묻기 허용 (GDR4-2)
- **D-031** 능동 제안 = 잠정 칩, 확정 전 diff 금지 (GDR4-3)

## 인계
- T2(schema): GDR4-2 채택 시 EngineResponse에 `mode` 분기 추가 — D-001/D-002 무영향 점검.
- T3(engine): GDR4-1 결정이 전송 계층을 가름. B 선택 시 SSE 신설.
- T4(ui): 스트리밍 렌더 + ask 모드 버블 + 잠정 칩 + 근거 패널.
- 비범위: COOK/POSTMORTEM/저장/로그인.

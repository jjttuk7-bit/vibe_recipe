# PRD.md — 제품 요구사항 정의

## 1. 문제

레시피 앱은 레드오션이지만 전부 **읽기 전용 매체**다 — 요리책의 디지털 버전. 백종원도, 만개의레시피도, ChatGPT도 레시피를 주고 손을 뗀다. 사용자가 실제로 요리한 결과("이거 짜다", "3번에서 탔다")는 별점으로 죽고 나에게도 다음 사람에게도 반영되지 않는다.

## 2. 해결

레시피를 **실행 가능한 프로그램**으로, 부엌을 **런타임 환경**으로, 사용자를 **자기 부엌을 디버깅하는 사람**으로 재정의한다.

> 바이브 코딩이 "코드를 몰라도 만들 수 있다"였다면,
> 바이브 레시피는 "요리를 몰라도 내 부엌에 맞춰 고쳐나갈 수 있다"이다.

## 3. 타깃 사용자

- 1차: 자취/1인가구, 냉장고 재료로 뭘 해먹을지 막막한 사람. 레시피대로 했는데 자꾸 실패하는 사람.
- 특성: 모바일 우선, 손에 양념 묻은 채 요리, 정밀 계량보다 "내 입맛/내 화구"에 맞추고 싶음.

## 4. 핵심 사용자 여정

1. 냉장고 재료를 던진다 → AI가 방향 2~3개 제안 (BUILD)
2. 한 단계씩 합의하며 레시피 완성 → ✓ build success
3. "요리 시작" → 스텝별 타이머, 핸즈프리 진행 (COOK)
4. 조리 중 "너무 짜요" → 즉석 핫픽스 (COOK 인라인 디버그)
5. 완성 → "어땠어?" 30초 회고 (POSTMORTEM)
6. 다음에 같은/비슷한 레시피를 빌드하면, 지난 실패가 미리 반영됨 (FINGERPRINT 주입)

## 5. 기능 요구사항

### F-1. BUILD MODE (구현 완료, v3)
- 대화형 점진 빌드 (한 턴 한 단계)
- 생성은 산출물 카드, 수정만 diff (접기 지원)
- 빌드 파이프라인 시각화, 선택지 칩, 패치 규율, warnings(린트)
- "알아서 다 해줘" 즉시 빌드 탈출구

### F-2. COOK MODE (신규 — 핵심)
- 스텝 단위 진행 (한 번에 한 스텝 크게)
- 스텝 내장 타이머 (`timer_sec` 기반, 백그라운드 알림)
- 핸즈프리: 큰 버튼, Wake Lock(화면 켜짐 유지)
- 인라인 핫픽스: 조리 중 문제 → 현재 스텝 기준 응급 패치 (레시피 불변, D-006)

### F-3. POSTMORTEM (신규)
- 3단 결과 평가 (좋음/그냥/망함)
- 실패 시 스텝 핀포인트 ("몇 번에서?")
- 결과는 CookRun으로 저장 → RuntimeLog 갱신

### F-4. FINGERPRINT (신규 — 해자)
- 여러 레시피의 RuntimeLog를 사람별로 집계
- "이 부엌: 화력 강함 / 면류 주의 / 단맛 회피" 프로파일
- 다음 BUILD 첫 프롬프트에 주입 (회귀 방지)
- 사용자에게 프로필로 노출 (전환 비용 형성)

## 6. 비기능 요구사항
- 모바일 우선 (조리 중 사용)
- 응답 지연 최소화 (Haiku 기본)
- 데이터 영속: Supabase (localStorage 금지 — 해자 자산)
- 보안: API 키 서버 격리, `/api/*` rate limit (배포 전 P0)

## 7. 성공 지표 (초기)
- 활성화: BUILD 완료 → COOK 진입률
- 핵심: COOK 완료 → POSTMORTEM 기록률 (용접이 작동하는지)
- 리텐션: 2회차 이상 요리 비율, Fingerprint 주입이 실패율을 낮추는지

## 8. 범위 밖 (Phase 2+)
- 음성 진행 (Web Speech API)
- 멀티 타이머 동시 진행
- 집단 부엌 지성 (네트워크 효과, 사용자 임계점 이후)
- 레시피/Fingerprint 커뮤니티 공유

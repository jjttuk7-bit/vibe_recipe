# TASTE.md — 취향/심미안 원칙 (해자 D-009)

> 이 문서는 명세서로 못 옮기는 판단들을 모은다. 유케이의 커피·차 도메인 감각(로스팅, 관능평가)에서 온다.
> Claude Code에게: 여기 없는 새 판단이 필요하면 임의로 결정하지 말고 유케이에게 물어라. 이 판단들이 곧 복제 불가능성이다.
> 이 문서는 살아있다. 새 결정이 나올 때마다 추가한다.

---

## 1. 맛을 어떻게 분해하는가

### 맛 6축 (왜 이 6개인가)
`매운맛 / 짠맛 / 단맛 / 신맛 / 감칠맛 / 기름짐`

- 기본 5미(단·짠·신·쓴·감칠)에서 **쓴맛을 빼고** 매운맛·기름짐을 넣었다. 이유: 가정 요리에서 사용자가 *조절하려는* 축만 남긴다. 쓴맛은 조절 대상이 아니라 대개 실패의 결과(탐)다 → warnings/RuntimeLog로 다룰 일이지 게이지가 아니다.
- 매운맛은 한국 가정식에서 독립 조절축이라 분리. 기름짐(fatty)은 미각이 아니라 입안 감각(mouthfeel)이지만, 사용자가 "느끼해"로 가장 자주 조절하는 축이라 맛에 포함.

### 식감 5축
`바삭 / 부드러움 / 쫄깃 / 국물 / 걸쭉`

- 식감은 맛보다 *조리 방식*에 직결된다. "바삭"을 올리면 굽기/튀기기로, "국물"을 올리면 끓이기로 조리법 자체가 바뀐다. 그래서 식감 게이지는 단순 양념이 아니라 *조리법 선택*의 입력이다.

### 게이지를 다룰 때
- 절대값보다 **변화량**이 사용자에게 의미 있다. "짠맛 7"보다 "짠맛 +2"가 직관적. (diff가 변화량을 보여주는 이유)
- 한 축을 올리면 물리적으로 따라 움직이는 축이 있다 (예: 기름짐↑ → 감칠맛 체감↑). 엔진은 이를 반영하되 사용자가 명시 안 한 축은 과하게 건들지 않는다 (패치 규율).

## 2. 스텝을 어떻게 자르는가

- **최대 6스텝.** 가정 요리에서 7스텝을 넘으면 사용자가 길을 잃는다. 복잡한 요리는 스텝을 합쳐 추상화한다 ("양념 재료를 모두 섞는다"로 묶기).
- 각 스텝은 **한 동작 + 한 판단 기준**. "중불에 3분"이 아니라 "중불에서 가장자리가 갈색이 될 때까지 (약 3분)". 시간은 보조, 감각 신호가 주(主). 단 `timer_sec`는 별도 필드로 항상 제공(D-005).
- **핵심 스텝 1개를 명시한다.** 모든 레시피엔 성패를 가르는 한 스텝이 있다 (예: "삼겹살 기름 먼저 뽑기"). 빌드 완료 시 이것만 말한다 (축하 금지).

## 3. 핫픽스 우선순위 (조리 중 응급 처치)

문제 발생 시 제안 순서의 원칙: **되돌릴 수 있는 것부터, 적게 개입하는 것부터.**

| 증상 | 1순위 (저개입) | 비고 |
|---|---|---|
| 너무 짜다 | 물/육수 추가, 불 줄이기 | 소금은 못 빼지만 희석은 됨. 면이 있으면 면이 흡수 |
| 싱겁다 | 소금보다 **감칠맛원**(된장/액젓) 먼저 | 짠맛만 올리면 밋밋. 감칠맛이 같이 올라야 맛이 "선다" |
| 탄다 | 즉시 불 끄고 팬 내리기 → 탄 부분 격리 | 긁지 말 것 (쓴맛 번짐) |
| 묽다 | 강불로 졸이기 (면은 먼저 건지기) | 전분물은 최후수단 (질감 변함) |

원칙: 양념을 *더 넣는* 해결책은 항상 마지막. 가정 요리 실패의 대부분은 "이미 너무 많이 넣어서"다.

## 4. 언어 톤

- 페어 셰프는 친구지 선생이 아니다. "~하세요"보다 "~하면 돼". 단 위험할 땐 단호하게 ("즉시 불 끄세요").
- 전문용어는 괄호로 풀어준다. "유화(기름과 물이 섞이는 것)".
- 칭찬 인플레이션 금지. 빌드 완료 시 "완성!"은 한 번이면 족하다. 매 스텝 칭찬하지 않는다.

# VIBE RECIPE 2.0 — "Cook = Run"

> 세상 모든 레시피 앱은 레시피를 주고 손을 뗀다.
> 우리는 부엌에서 일어난 일을 시스템이 기억하게 만든다.

---

## 0. 한 줄 정의

**레시피를 코드처럼 빌드하고(write), 부엌에서 실행하고(run), 런타임 에러를 다음 빌드로 되먹이는(debug) 페어 쿠킹 IDE.**

기존 v3가 "레시피 생성기"였다면, 2.0은 요리를 **컴파일이 아니라 런타임**으로 다룬다. 코드를 작성만 하고 실행하지 않는 개발자는 없다. 그런데 모든 레시피 앱은 정확히 그렇게 동작한다 — 작성하고 끝.

---

## 1. 왜 기존에 없는가

| | 백종원/만개의레시피 | ChatGPT 레시피 | VIBE v3 (우리 이전 버전) | **VIBE 2.0** |
|---|---|---|---|---|
| 레시피 생성 | 고정 DB | 1회 답변 | 대화형 빌드 + diff | 대화형 빌드 + diff |
| 수정 | 불가 | 전체 재생성 | 패치 (부분 수정) | 패치 |
| **실제 조리 진행** | 글만 읽음 | 글만 읽음 | ❌ | **Cook Mode (핸즈프리)** |
| **조리 결과 기록** | ❌ | ❌ | ❌ | **Runtime Log** |
| **결과→레시피 피드백** | ❌ | ❌ | ❌ | **자동 회귀 주입** |

핵심 공백: **요리한 결과가 레시피로 되돌아오는 제품이 존재하지 않는다.** "이 레시피 짜다"는 리뷰는 별점으로 죽고, 다음 사람에게도 나에게도 반영되지 않는다.

## 2. 세 가지 모드 (= 개발 사이클)

### ① BUILD MODE (이미 구현됨, v3)
레시피를 대화로 빌드. concept → base → taste → steps → done.
생성은 산출물 카드로, 수정만 diff로.

### ② COOK MODE (신규 — 핵심)
`recipe.run()`. 빌드된 레시피를 부엌에서 실제로 실행하는 모드.

- **스텝 단위 진행**: 한 번에 한 스텝만 크게 표시. 다음 스텝은 탭 또는 음성.
- **스텝 내장 타이머**: "중불 3분" 스텝이면 타이머가 자동으로 뜸. 백그라운드에서도 알림.
- **핸즈프리**: 손에 양념 묻은 채로 화면 못 만진다는 현실. 큰 버튼, 음성 "다음", 화면 켜짐 유지.
- **인라인 디버그 (핫 리로드)**: 조리 중 "어 너무 짜" 한마디 → AI가 *현재 스텝 기준* 복구 액션을 즉시 삽입. 레시피 전체를 다시 안 짠다. 물 추가, 불 줄이기 같은 응급 패치.

### ③ DEBUG / POSTMORTEM (신규)
조리가 끝나면 30초 회고. 이게 런타임 로그가 된다.

- "어땠어?" → 😋 좋음 / 😐 그냥 / 😵 망함
- 망했으면 **어느 스텝에서** 터졌는지 핀포인트: "3번에서 탔음", "5번 소스가 너무 졸았음"
- 이 로그가 `RuntimeLog`로 레시피에 영구 첨부됨.

## 3. 피드백 루프 (이 제품의 심장)

```
BUILD ──> COOK ──> POSTMORTEM
  ▲                      │
  └──── RuntimeLog 주입 ──┘
```

같은 레시피를 다시 빌드/요리할 때, 지난 런타임 로그가 자동으로 엔진에 주입된다.

> 예: 지난번 "3번 스텝에서 탔음" 로그가 있으면 →
> 다음 빌드 때 AI가 먼저 말한다: *"지난번에 3번에서 태웠으니, 이번엔 불을 중약불로 낮추고 30초 줄일게."*

이것이 **회귀 테스트**다. 한 번 고친 버그는 다시 안 난다. 요리를 거듭할수록 그 사람의 부엌(화력 센 가스레인지, 얕은 팬)에 맞게 레시피가 수렴한다 — **개인화의 진짜 정의**.

## 4. 데이터 모델 추가분

기존 `RecipeState`는 그대로 두고, 두 개를 추가한다.

```ts
// 조리 1회 = 1 run
type CookRun = {
  recipe_id: string;
  started_at: string;
  completed: boolean;
  outcome: "good" | "meh" | "failed" | null;
  step_events: {
    step_index: number;
    type: "done" | "timer_done" | "hotfix" | "failed_here";
    note?: string;        // "너무 짜서 물 추가", "여기서 탔음"
    timestamp: string;
  }[];
};

// 레시피에 누적되는 런타임 지식
type RuntimeLog = {
  recipe_id: string;
  total_runs: number;
  known_issues: {            // 회귀 방지 목록
    step_index: number;
    issue: string;           // "3번 강불이면 탐"
    fix_applied?: string;    // "중약불 + 30초 단축으로 해결"
    resolved: boolean;
  }[];
};
```

엔진 프롬프트에는 빌드 시작 시 해당 레시피의 `RuntimeLog.known_issues`를 주입한다.

## 6. MVP 범위 (주말 1개 분량)

포함:
- Cook Mode: 스텝 진행 + 타이머 + Wake Lock(화면 켜짐) + 핫픽스(인라인 디버그)
- Postmortem: 3단 평가 + 스텝 핀포인트
- RuntimeLog: localStorage가 아니라 Supabase (개인화의 자산이므로 영속 필수)
- 다음 빌드 시 known_issues 주입

제외 (Phase 2):
- 음성 진행 ("다음" / "헤이, 너무 짜") — Web Speech API
- 멀티 타이머 동시 진행
- 런타임 로그 커뮤니티 공유 (남의 부엌 데이터로 내 레시피 보정)

## 7. 기술 노트

- **Cook Mode의 타이머**는 스텝 텍스트에서 시간을 파싱하지 말 것. 빌드 단계에서 엔진이 각 스텝에 `timer_sec` 필드를 함께 출력하게 스키마를 확장한다 (텍스트 파싱은 환각·실패의 온상).
- **Wake Lock API**로 조리 중 화면 꺼짐 방지. 미지원 브라우저는 graceful degrade.
- **핫픽스는 새 버전을 만들지 않는다.** 조리 중 응급 조치는 `CookRun.step_events`에만 기록되고, 정식 레시피 수정은 Postmortem 이후 사용자가 "다음엔 이렇게" 할 때만 BUILD diff로 승격.
- **RuntimeLog 주입 위치**는 빌드 첫 턴의 시스템 컨텍스트. "이 사용자의 이 레시피는 과거에 ○○에서 실패했다"를 알고 시작.

---

## 8. 이게 왜 중요한가 (포지셔닝)

레시피 앱은 레드오션이다. 하지만 전부 **읽기 전용 매체**다. 요리책의 디지털 버전일 뿐.

VIBE 2.0은 레시피를 **실행 가능한 프로그램**으로, 부엌을 **런타임 환경**으로, 사용자를 **자기 부엌을 디버깅하는 사람**으로 재정의한다. 더 이상 "남의 레시피를 따라 하는" 것이 아니라 "내 부엌에 맞게 레시피를 수렴시키는" 경험.

> 바이브 코딩이 "코드를 몰라도 만들 수 있다"였다면,
> 바이브 레시피 2.0은 "요리를 몰라도 *내 부엌에 맞춰 고쳐나갈* 수 있다"이다.

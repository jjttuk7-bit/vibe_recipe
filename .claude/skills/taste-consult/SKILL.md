# Taste Consult — 도메인 판단 컨설팅 + TASTE.md 등재

이 스킬은 `doc-taste-scribe` 에이전트가 사용하는 표준 절차다. 핵심 원칙은 **"임의 결정 없음"**이다. 결정은 항상 사용자(유케이)가 내리고, scribe는 결정의 일반화와 등재만 담당한다.

## 트리거

다음 시그널이 오면 이 스킬을 적용한다:

- 다른 에이전트로부터 SendMessage로 "TASTE 컨설팅 요청"
- 코드 작성 중 명세화되지 않은 미세 도메인 판단이 필요한 경우
- 기존 TASTE.md 원칙으로는 케이스가 모호한 경우

## 입력 / 출력

- 입력: 컨설팅 요청 (컨텍스트 + 무엇을 결정해야 하는지)
- 출력:
  - 리더(오케스트레이터)에게 SendMessage: 사용자에게 물을 질문 본문
  - 결정 후 `docs/TASTE.md` 갱신 + 요청자에게 결정 통보

## 4단계 절차

### Step 1: 기존 TASTE.md 정밀 조회

먼저 TASTE.md에 정말 없는지 확인한다. 비슷한 영역의 원칙이 있으면 그 원칙으로 충분한지부터 검토.

- 인접 원칙이 있고 정확히 적용 가능 → 등재 불필요. 요청자에게 인접 원칙 인용하여 회신.
- 인접 원칙이 있지만 케이스가 모호 → Step 2로 진행 (원칙 확장)
- 완전히 새 영역 → Step 2로 진행 (신규 등재)

### Step 2: 후보 구성 (2~3개)

사용자에게 묻기 전에 충분한 컨텍스트로 후보를 정리한다. 막연한 "어떻게 할까요?" 금지.

각 후보에 다음을 포함:
- **무엇**: 후보의 핵심 한 줄
- **사유**: 도메인적으로 왜 이 선택이 말이 되는지
- **트레이드오프**: 어떤 손해가 따라오는지
- **재사용**: 이 결정이 일반화되어 어디서 또 쓰일지

예시(맛 게이지 항목 수):
- 후보 A: 5개 (단/짠/신/감칠/매) — 단순. 한국 사용자에게 직관적. 트레이드오프: 향·식감 표현 불가.
- 후보 B: 6개 (5개 + 식감) — 식감을 분리. 트레이드오프: 라디오 그리드 UI에서 한 행에 안 들어감.
- 후보 C: 9개 (전문가식 분해) — 정밀. 트레이드오프: 일반 사용자에게 과하다, 회고 30초 실패.

### Step 3: 리더에게 사용자 질문 트리거

SendMessage로 리더에게 다음 형식으로 요청:

```markdown
## 사용자 결정 요청 — {결정 영역}

**컨텍스트**: 어디서 왜 결정이 필요한가 (요청 에이전트, 어느 작업)

**질문**: {한 줄 질문}

**후보**:
1. **A**: {한 줄} — 트레이드오프: {요약}
2. **B**: {한 줄} — 트레이드오프: {요약}
3. **C**: {한 줄} — 트레이드오프: {요약}

**일반화 메모**: 이 결정은 향후 {유사 상황}에도 적용될 수 있도록 원칙 형태로 TASTE.md에 등재할 예정입니다.
```

리더는 사용자에게 AskUserQuestion 또는 직접 질문 형태로 전달한다.

### Step 4: 결정 등재 (일반화 + 요청자 통보)

사용자가 결정하면:

1. **원칙으로 일반화**: 단순히 "맛 게이지는 6개"가 아니라, "회고 입력은 30초 내 완료를 우선한다. UI 그리드 한 행에 들어가는 최대 N개를 상한으로 둔다." 같이 다른 케이스에도 적용 가능한 원칙으로 변환.

2. **TASTE.md 등재**: 적절한 섹션에 다음 형식으로 추가:
   ```markdown
   ### 원칙: {원칙명}

   **컨텍스트**: 어떤 상황에 적용
   **원칙**: {일반화된 결정}
   **이번 케이스**: {원래 결정 — 예: 맛 게이지 6개}
   **근거**: {왜 이 원칙이 도메인 감각에 부합하는지 — 사용자 답변에서 추출}
   **반대 사례**: {언제는 이 원칙이 안 통하는지 — 알려진 한계}
   ```

3. **요청자에게 통보**: SendMessage로 결정 + TASTE.md 섹션 참조 전송. 요청자가 작업 재개.

4. **CLAUDE.md 변경 이력**: 새 TASTE 원칙 등재를 변경 이력에 짧게 기록.

## 회피해야 할 안티 패턴

- **임의 디폴트 사용**: "일단 6개로 가고 나중에 바꿔도 됨" → 금지. D-009 위반. 묻기.
- **기술적 후보로 환원**: "JSON 필드 6개 vs 9개" → 도메인적 후보(맛 인지의 단순성 vs 정밀성)로 표현해야 사용자가 판단 가능.
- **하나의 후보만 제시**: 사용자가 "그래 그렇게 해"라고만 답하게 만드는 질문. 후보 2개 이상.
- **너무 추상적인 일반화**: "맛 관련은 사용자 친화적으로" → 적용 불가능. 구체적이되 케이스 일반화.
- **즉답 강요**: 사용자가 즉답 못 하면 "추후 결정" + 가장 보수적 옵션 + TASTE.md에 보류 표시.

## 사용 예시

**요청 (engine-builder로부터)**: "lib/prompt.ts에서 식감 분류 후보를 정의해야 합니다. TASTE.md에 식감 분류 원칙이 없어 컨설팅 요청합니다."

**Step 1 (조회)**: TASTE.md에 "맛 게이지 원칙"은 있으나 "식감 분류"는 없음. 인접 원칙은 식감을 다루지 않음 → 신규 등재 필요.

**Step 2 (후보 구성)**:
- A: 4분류 (바삭/쫀득/부드러움/거침) — 사용자 표현에 가까움
- B: 7분류 (전문 관능평가식) — 정밀하나 일반 사용자에게 과함
- C: 3분류 (좋음/그냥/싫음) + 자유 메모 — 회고 부담 최소

**Step 3 (리더에게)**: 위 후보를 SendMessage로 리더에게 전달, 사용자 질문 요청.

**Step 4 (등재)**: 사용자가 A 선택 → "회고 분류는 사용자 어휘 4~5개 + 필요 시 자유 메모. 전문가 어휘는 도입하지 않는다."로 일반화하여 TASTE.md 등재. engine-builder에게 결정 통보.
